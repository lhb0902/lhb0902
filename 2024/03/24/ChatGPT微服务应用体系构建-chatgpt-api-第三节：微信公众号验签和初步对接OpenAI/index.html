4. 相关知识

1. DDD架构（领域驱动设计）分层

  • 用户接口层（User Interface层）：负责与用户交互，显示信息并解释用户指令，这可能包括用户、程序、自动化测试和批处理脚本等；
  • 应用层（Application层）：位于用户接口层和领域层之间，主要定义软件要完成的任务，并协调表达领域概念的对象来解决问题。应用层应尽量简单，不包含业务规则或知识，而是专注于协调领域对象之间的任务和协作；
  • 领域层（Domain层）：这是业务逻辑的核心，负责表达业务概念、业务状态和业务规则。领域层包含实体、值对象、领域服务等领域模型中的对象；
  • 基础层（Infrastructure层）：位于所有层之下，提供通用的技术和基础服务，如工具、驱动、消息中间件、网关、文件、数据库等；

2. MVC（Model-View-Controller）架构分层

  • 模型model：编写程序应用功能（实现算法等等），数据库管理，也就是实体类。用于数据的封装和数据的传输；
  • 视图view：界面设计人员进行图形界面设计，也就是GUI组件，用于数据的展示；
  • 控制器 controller：对请求进行处理，负责请求转发，也就是事件，用于流程的控制；

3. SSM（Spring、SpringMVC、MyBatis）架构分层

三个框架的作用：

  • SpringMVC：作为 View 层的实现者，完成用户的请求接收功能。SpringMVC 的 Controller作为整个应用的控制器，完成用户请求的转发及对用户的响应；
  • MyBatis：作为 Dao 层的实现者，完成对数据库的增、删、改、查功能；
  • Spring：以整个应用大管家的身份出现。整个应用中所有 Bean 的生命周期行为，均由Spring 来管理。即整个应用中所有对象的创建、初始化、销毁，及对象间关联关系的维护，均由 Spring 进行管理；

架构中各层的作用：

  • 持久层（Dao）：DAO层主要是做数据持久层的工作，负责与数据库进行联络的一些任务都封装在此；
  • 业务层（Service）： Service层主要负责业务模块的逻辑应用设计；
  • 表现层（Controller/Handler）：Controller层负责具体的业务模块流程的控制，在此层里面要调用Service层的接口来控制业务流程；
  • 视图层（Viewer）：Viewer层主要负责页面显示，用户交互等；
